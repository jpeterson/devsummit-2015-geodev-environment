Automation lets you remove the tedious and repetetive steps from your life. Stop working so hard.
Grunt and Gulp are the two foremost Node-based taskrunners, both have their pros and cons, fanboys and haters... but both do great things.
npm to install plugins package.json to manage them

Grunt and all it's plugins are Node packages, so we use NPM to install (use --save-dev to save them to your package.json).

Remember to save to config files, so all devs are sure to be running the same environment.

Configure tasks in Gruntfile.js

grunt-contrib-connect
start a web server
grunt-concurrent
run tasks concurrently

grunt-contrib-connect gives you a server in your repo. This helps env consistency on a team. Also opens up server to manipulation through grunt (i.e. live reload).

grunt-concurrent is purely a time-saver, but shaving seconds off many repetitive tasks adds up.

Automation for Python

Task automation for Python GeoDevs is less standardized and mainstream. We would like to share one method that works for us.
As we all know, to publish arcpy GP tasks to server, you have to have a successful result, then you can publish. That really puts the block on "automation".

The Plan*

  1. Use Python Toolbox
  2. Grunt task to copy files from working directory to: 
    arcgissystem\arcgisinput\[folder]\[tool name].GPServer\extracted\v101\
  3. Grunt task to restart ArcGIS Server: grunt-ags-admin
* Use for development only, and at your own risk!
After this, short demo of how this works.
Initial task. Grunt task that is watching our "src" folder, and when it sees a changes, runs grunt sync to copy the files to our AGS config store, and then restarts the service using the grunt-ags-admin grunt task. So for a quick demo, lets add in a single line that will log out a message to the output of the GP service. When we save the file, the grunt task will run, copying the files and restarting the service. Then we'll run the GP service again through our REST endpoint, and BAM it's automatically updated!
